# AGENTS.md - 《未名舟》Codex 创作约定

## 项目性质

这是长篇中文网文《未名舟》/ 命名权世界版重构稿项目，不是代码项目。

正文只写入 `chapters_txt/第一卷_地球舟材/*.txt`。正文文件保持纯 TXT，不写 Markdown 标题、表格、审计备注或创作术语。

## 默认入口

所有继续、重写、审稿、发布前检查、五章提交、十章复盘，都先按 `kid3-core-workflow` skill 的入口流处理。项目文档负责记忆，skill 负责流程和软审。

不确定该读哪份文档时，先看 `docs/README_文档使用索引.md`，不要默认把 `docs/` 全量加载进上下文。

## 快捷别名

- `继续` / `继续创作`：按普通续写加载，读最新尾巴、账本、伏笔索引和当前阶段口径；完成正文、轻审、小修和状态回写。
- `审`：默认做软审，只抓最影响追读和听感的 1-3 个点；用户要求发布前再跑完整检查。
- `查[关键词]`：优先用 `rg` 搜当前卷正文和必要文档，只返回相关匹配，不全文重读。
- `节点`：处理五章发布包、十章复核或五十章卷审，不把未到批次的新章混进去。

## 普通续写加载

用户只说“继续/继续创作/下一章”时，默认读取：

- `git status --short --branch`
- 最新 1-2 章结尾
- `docs/14_当前故事状态账本.md`
- `docs/伏笔索引.md`
- 当前阶段补纲，以 `docs/README_文档使用索引.md` 标注为准；已完成阶段补纲只在回查时读
- `docs/创作复盘_常见问题与改进.md` 中最近且相关的 1-3 条

如果最新章节已经到五章、十章或五十章节点，先处理节点，再继续新章。若新章已经越过节点，旧节点按原范围处理，不把新章混进旧批次。

## 状态回写

新章节完成后，若出现以下变化，顺手更新 `docs/14_当前故事状态账本.md`：

- 最新完成章节
- 明面主冲突或章末物理钩
- 角色状态、伤、能力消耗、旧物位置
- 敌方新经济商品、文书、样本、税、榜单
- 未结算的旧账折痕

状态账本只记事实和短期牵引，不写写作课，不替代大纲。

伏笔位置、旧债状态和敌方样本索引写入 `docs/伏笔索引.md`。阶段完成后，把已经不适合日常入口的细节折入 `docs/阶段快照_*.md`，不要让状态账本重新长成逐章大纲。

回写时也要顺手清掉已经完成的“下一步/可转成/后续可以”旧措辞，避免状态账本保留过期箭头。

## 软审口径

审计结果只当“看一眼这里”的证据，不当硬判定。破墙词、AI腔、工艺词、字数、章节钩、听感、读感，都用软落地方式处理。

正文优先保证场景自然流动。热场可以直白，冷场可以留白，有旧物时再给纹理；不要为了显得文学而把每一句都磨亮。

软手感三句：

1. 先让人疼、渴、亏或做选择，再告诉读者规则叫什么。
2. 讲理不能连过三句；第四句尽量换成动作、物件变化、敌人提前动手或一句人话。
3. 章末落在物理战利品、伤口、旧物、敌方新追口或船体变化上，不落在“大家理解了规则”。

## Git 习惯

不默认单章提交。通常每五章形成一个提交批次，先跑发布包和轻审，再 stage/commit/push。若用户明确说“提交”，按当前已完成范围处理，并说明是否混有未到批次的新章。
