# AGENTS.md - 《未名舟》Codex 创作约定

## 项目性质

这是长篇中文网文《未名舟》/ 命名权世界版重构稿项目，不是代码项目。

正文只写入 `chapters_txt/第一卷_地球舟材/*.txt`。正文文件保持纯 TXT，不写 Markdown 标题、表格、审计备注或创作术语。

## 默认入口

所有继续、重写、审稿、发布前检查、五章提交、十章复盘，都先按 `weimingzhou-novel-writer` skill 的入口流处理。项目文档负责记忆，skill 负责流程和软审。

不确定该读哪份文档时，先看 `docs/README_文档使用索引.md`，不要默认把 `docs/` 全量加载进上下文。

## 普通续写加载

用户只说“继续/继续创作/下一章”时，默认读取：

- `git status --short --branch`
- 最新 1-2 章结尾
- `docs/14_当前故事状态账本.md`
- 当前阶段补纲，以 `docs/README_文档使用索引.md` 标注为准，例如 `docs/26_051-060阶段补纲_退约牌与白灯夜航.md`
- `docs/创作复盘_常见问题与改进.md` 中最近且相关的 1-3 条

如果最新章节已经到五章、十章或五十章节点，先处理节点，再继续新章。若新章已经越过节点，旧节点按原范围处理，不把新章混进旧批次。

## 状态回写

新章节完成后，若出现以下变化，顺手更新 `docs/14_当前故事状态账本.md`：

- 最新完成章节
- 明面主冲突或章末物理钩
- 角色状态、伤、能力消耗、旧物位置
- 敌方新经济商品、文书、样本、税、榜单
- 未结算的旧账折痕

状态账本只记事实和短期牵引，不写写作课，不替代大纲。

回写时也要顺手清掉已经完成的“下一步/可转成/后续可以”旧措辞，避免状态账本保留过期箭头。

## 软审口径

审计结果只当“看一眼这里”的证据，不当硬判定。破墙词、AI腔、工艺词、字数、章节钩、听感、读感，都用软落地方式处理。

正文优先保证场景自然流动。热场可以直白，冷场可以留白，有旧物时再给纹理；不要为了显得文学而把每一句都磨亮。

## Git 习惯

不默认单章提交。通常每五章形成一个提交批次，先跑发布包和轻审，再 stage/commit/push。若用户明确说“提交”，按当前已完成范围处理，并说明是否混有未到批次的新章。
